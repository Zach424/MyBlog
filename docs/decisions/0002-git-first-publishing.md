# ADR 0002：Git-first 自助发布

- 状态：accepted
- 日期：2026-07-19

## 背景

博客已经以仓库内 Markdown 为内容源，但日常发布仍需要人工完成构建、Sites 打包和部署。作者要求网页发布不依赖 Codex，并希望用 Obsidian 编辑同一批文章。

## 决策

GitHub 仓库继续作为文章、项目、附件、历史和回滚的唯一事实源。提供两个等价编辑入口：`/studio` 网页后台和本地 Obsidian Vault。两者都只产生 Git 提交；main 上的精确提交由 GitHub Actions 检查并发布到作者自己的 Cloudflare 账号。

第一阶段不使用 D1 保存文章，也不实现 Git 与数据库双向同步。图片先保存在仓库内；只有附件规模触发明确的仓库或构建成本问题后，才评估 R2。

## 原因

- 保留现有 Markdown 内容契约、稳定 URL、Git 历史和回滚模型；
- 网页后台与 Obsidian 不会争夺“最新版本”；
- 没有数据库迁移、导出和双向同步故障面；
- 发布失败会停在 GitHub 质量门，线上继续运行最近一次成功版本；
- Cloudflare 凭证属于作者的 GitHub Environment，不进入源码和轮次档案。

## 代价与约束

- 发布需要一次 Git 提交与构建，不能像数据库后台一样瞬时修改生产数据；
- GitHub OAuth 应用和 Cloudflare secrets 需要作者首次配置；
- 同时在网页后台与 Obsidian 编辑同一文件可能产生 Git 冲突，发布前必须先同步；
- 大型二进制附件未来可能需要迁移到 R2，但当前不提前增加该复杂度。

## 被拒绝的方案

以 D1 作为第二内容源会迫使系统定义 Git 与数据库谁拥有最终写入、如何合并冲突、如何让 Obsidian 离线编辑以及如何回滚正文。当前真实需求可以由单一 Git 内容源完成，因此不采用。
