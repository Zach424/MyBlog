## 6. 实现方法

先读取当前 Sites 项目，确认访问模式为 `custom` 且平台允许切换为 `public`；在用户明确授权后清空非必要共享范围并更新访问策略。权限变更立即作用于已经验收过的生产版本，因此先不修改源码，用无凭证 HTTP 和未登录浏览器验证访问边界，避免把代码变化与权限问题混在一起。

HTTP 验收直接请求关键端点，再解析生产 Sitemap 并逐一访问全部 URL。浏览器会话清空 Cookie 与 Local Storage，确认页面没有登录门禁；随后缩到 320 × 568，比较 `scrollWidth` 与 `clientWidth`，再从主导航进入搜索并输入 `Cloudflare`。通过这些证据后才更新 REV.010、项目文档和维护流程，并用同一提交构建、保存和发布最终同步版本。

## 8. 经验与教训

“部署到生产”与“公众可以访问”是两个独立状态。私有生产版本可以完整验证代码和 Worker 行为，却不能证明陌生访客不会被权限层拦截。访问策略一旦变化，最小验收单位应从已登录所有者切换为无凭证请求与全新浏览器状态。

先切权限、验证旧版本，再修改展示状态，可以把故障边界分开：如果无凭证访问失败，问题在托管访问策略；如果 REV.010 发布后失败，问题更可能在新提交或构建。这个顺序比把权限、代码和内容一次性发布更容易回滚和解释。

公开并不意味着自动发布所有个人信息。博客、源码、邮箱、统计和评论是不同的公开边界，应该分别由实际需求和明确授权决定。
