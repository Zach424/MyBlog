本轮只冻结文章、TIL、专题、标签和项目的输入输出契约，不修改产品页面或选择渲染库。

成功标准：字段、稳定 URL、跨内容约束和 Cloudflare 构建边界明确；文档内部链接有效；变更形成独立提交并推送。

## 3. 设计内容

本轮不改变视觉基线。Commit Trace、Knowledge Index、Workbench Signals 三个视觉候选已经在源码目录外生成，等待下一轮用户选择；在选择前不修改正式产品页面。

## 4. 使用的技术

- Markdown frontmatter 作为内容输入格式；
- Vite `import.meta.glob` 作为目标打包边界；
- 构建期 schema 校验和 Markdown 渲染；
- Git 文件名作为稳定 slug 来源。

## 5. 实现的功能

- 冻结文章与 TIL 的必填、可选字段；
- 冻结项目状态、技术栈、仓库和演示字段；
- 从文章的 `series`、`tags` 派生专题与标签索引；
- 规定草稿、未来日期、精选内容和 URL 的跨内容校验；
- 规定生产运行时不得依赖文件系统读取。

## 6. 实现方法

先从长期内容语义和稳定 URL 出发设计契约，再把解析库推迟到实现轮选择。这样未来替换 Markdown 工具时，文章元数据和公开 URL 不需要跟着迁移。

## 7. 验证证据

- Markdown 本地链接检查：通过，`docs` 下相对链接目标全部存在。
- 产品源代码和依赖未变化，因此不重复运行与本轮无关的浏览器验收；基线构建证据沿用 iteration 0001。

## 8. 经验与教训

- 文件名 slug 比发布日期路径更稳定，文章更新时间不会改变公开 URL。
- 专题和标签应从文章字段派生，第一版不需要单独数据库。
- 先冻结输入输出再挑选解析库，可以避免工具 API 反向塑造内容结构。
- 视觉选择和内容契约可以拆成两个提交，减少等待用户决策时的悬空改动。
