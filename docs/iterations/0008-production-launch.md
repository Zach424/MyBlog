# Iteration 0008：Cloudflare 生产上线

## 1. 范围与成功标准

本轮唯一目标是把第 0007 轮发布候选部署到 Sites / Cloudflare，并以真实生产请求验证全部公开路由、发现端点、结构化数据、缓存和安全头。只有在线验收通过后，项目才从 `building` 更新为已上线状态。

## 2. 项目结构状态

首次部署写入 `.openai/hosting.json` 的 Sites 项目标识，D1 与 R2 继续为 `null`。在线故障修复新增 16 个薄静态包装页：3 个文章、1 个项目、1 个专题和 11 个标签路径；原 `[slug]` 页面继续集中维护读取、元数据、正文和 404 逻辑。

发布审计新增 Sitemap → `app/<public-path>/page.tsx` 覆盖检查。当前质量门为 12 项单元测试、7 项 Worker 集成测试和 6 项发布审计。

## 3. 设计内容

本轮不改变视觉方向。首页仍使用 Commit Trace 与 Evidence Rail，只把 Building 状态改为“Cloudflare 路由复核”，避免在详情路由尚未通过时提前宣称完整上线。显式包装页调用原详情组件，因此不产生第二套布局或视觉分支。

## 4. 使用的技术

- Sites 私有部署与版本化发布；
- Cloudflare Worker 生产日志；
- 私有站点旁路凭证，仅用于无界面 HTTP 验收，不写入仓库；
- Wrangler 本地 Workerd 与部署干跑；
- Next.js / Vinext App Router 的静态优先路由；
- Node test、Sitemap 驱动的文件覆盖审计。

## 5. 实现的功能

1. 创建仅所有者可访问、无群组共享的生产站点。
2. 保存并发布已验证构建，获得稳定 Sites URL。
3. 对首页、集合、详情、搜索、RSS、Sitemap、robots、图标和 404 发起真实生产请求。
4. 为所有公开详情路径增加显式静态包装，并保留原动态详情页作为单一实现来源。
5. 新增“公开详情 URL 必须有平台路由包装”的自动门槛。

## 6. 实现方法

严格按“构建 → 提交并推送同一源码 → 官方脚本打包 → 保存版本 → 私有部署 → 轮询结果 → 在线 HTTP 验收”执行。归档包包含 `dist/server/index.js`、静态资源和 `dist/.openai/hosting.json`，不包含凭证。

首次生产验收发现所有参数化详情路径 404 后，先用 Worker 日志确认请求已经进入应用且结果为应用级 404，再对同一构建执行 Node Worker 与本地 Wrangler Workerd 对照。两者都返回 200，说明内容与渲染逻辑正常，故障边界位于 Sites 生产参数化路由兼容层。

修复没有复制正文页面逻辑：每个公开 slug 的静态页只构造固定 `params`，继续调用 `[slug]/page.tsx` 的页面与元数据函数。静态路由在平台匹配中优先，动态页仍处理未列出的 URL。Sitemap 覆盖测试把这项兼容要求变成发布契约。

## 7. 验证证据

首次生产版本：部署状态成功；首页、关于、文章/项目/专题/标签集合、搜索、RSS 与 Sitemap 返回 200；文章、项目、专题和标签详情返回 404。Worker 日志显示请求 outcome 为 `ok` 且响应状态 404，不是平台崩溃或权限拒绝。

修复后本地证据：

- `npm run check`：Lint、12/12 单元测试、TypeScript、生产构建、7/7 Worker 测试、6/6 发布审计全部通过；
- 构建路由表列出 16 个显式公开详情路径，同时保留 4 个动态 sibling；
- Wrangler Workerd 按 Sitemap 请求 23/23 URL 全部返回 200，未知路径返回 404；
- Wrangler 部署干跑总上传 `2645.68 KiB`，gzip `625.60 KiB`，无 D1/R2 绑定；
- 当前尚待第二次生产发布与在线复核，因此本轮状态为 `partial`。

## 8. 经验与教训

“平台报告部署成功”只说明构建被接受和 Worker 已发布，不说明业务路由正确。上线门槛必须包含真实 URL 验收，尤其是集合页与详情页不能只抽查一类。

Node 中导入 Worker 的集成测试、Wrangler 本地 Workerd 和真实 Sites 生产仍可能出现不同结果。兼容层故障需要用三层证据缩小边界，不能仅凭本地通过推断生产正确。

生成静态参数不等于托管平台一定把参数化路由物化为显式路径。当前薄包装是一项有测试的兼容措施；上游修复后只有通过真实生产对照，才能安全删除。

首次故障还暴露了 404 HTML 会继承一小时边缘缓存。Worker 已改为只缓存成功 HTML，404 与其他 HTML 错误使用 `no-store`，避免瞬时路由错误被边缘放大。
