# Iteration 0008：Cloudflare 生产上线

## 1. 范围与成功标准

本轮唯一目标是把第 0007 轮发布候选部署到 Sites / Cloudflare，并以真实生产请求验证全部公开路由、发现端点、结构化数据、缓存和安全头。只有在线验收通过后，项目才从 `building` 更新为已上线状态。

## 2. 项目结构状态

部署已把 Sites 项目标识写入 `.openai/hosting.json`，D1 与 R2 继续为 `null`。内容管线新增 `build/content-build-date.ts`，Vite 在构建开始时按 `Asia/Shanghai` 生成 `__CONTENT_BUILD_DATE__`；`lib/content/index.ts` 使用该常量完成草稿与未来日期过滤，不再在 Worker 模块初始化时读取运行时时钟。

两次生产诊断之间曾增加 16 个显式详情路由包装；第二次生产证据证明详情 404 的根因是内容索引为空，而非参数化路由匹配，因此这些临时包装和对应测试已经撤回，没有把错误假设留在最终架构里。

## 3. 设计内容

本轮不改变视觉方向。首页继续使用 Commit Trace 与 Evidence Rail；在线验收通过后，Verified 更新为“私有生产上线”，Building 转向“公开访问与域名决策”，项目状态从 `building` 更新为 `maintained`。状态文案继续只表达已有证据，不把私有生产部署误写成公众可访问。

## 4. 使用的技术

- Sites 私有部署与版本化发布；
- Cloudflare Worker 生产日志；
- 私有站点旁路凭证，仅用于无界面 HTTP 验收，不写入仓库；
- Wrangler 本地 Workerd 与部署干跑；
- Vite `define` 注入构建常量；
- `Intl.DateTimeFormat.formatToParts` 与 `Asia/Shanghai` 作者时区；
- Node test 对 UTC → 作者日期边界的确定性验证。

## 5. 实现的功能

1. 创建仅所有者可访问、无群组共享的生产站点。
2. 保存并发布已验证构建，获得稳定 Sites URL。
3. 对首页、集合、详情、搜索、RSS、Sitemap、robots、图标和 404 发起真实生产请求。
4. 将公开内容日期从 Worker 模块级运行时时钟改为构建期作者时区常量。
5. 只缓存成功 HTML；404 与其他 HTML 错误响应使用 `Cache-Control: no-store`。

## 6. 实现方法

严格按“构建 → 提交并推送同一源码 → 官方脚本打包 → 保存版本 → 私有部署 → 轮询结果 → 在线 HTTP 验收”执行。归档包包含 `dist/server/index.js`、静态资源和 `dist/.openai/hosting.json`，不包含凭证。

首次生产验收看到详情 404，Node Worker 与本地 Wrangler Workerd 却都返回 200，初步假设是 Sites 参数化路由差异。显式静态包装发布后，生产 Sitemap 仍只有 7 个基础 URL，首页、文章集合和项目集合也完全没有内容；这条更强证据推翻了路由假设，说明详情 404 只是空内容索引的下游结果。

内容模块原先在模块初始化时调用 `new Date()`。在 Cloudflare 模块环境中，这不是可靠的内容发布基准。修复在 Vite 配置执行时计算作者日期，以字符串常量注入所有构建环境，再显式传给 `isPublished`。同一构建内的页面、搜索、RSS、Sitemap 和静态参数现在共享同一个日期快照。

## 8. 经验与教训

“平台报告部署成功”只说明构建被接受和 Worker 已发布，不说明业务内容正确。上线门槛必须检查页面正文和 Sitemap 数量，不能只看 HTTP 200。

详情 404 容易把注意力吸引到路由层，但集合页 200 不代表集合有数据。先比较输入索引、集合内容和详情结果，才能找到最早发生偏差的层级。

错误假设也应快速做成可证伪版本。显式包装发布后仍失败，证据足以立即删除它；保留“看似保险”的无效兼容层只会增加未来内容维护成本。

构建期内容应由构建输入决定。模块初始化时钟属于运行环境状态，会破坏同一提交在不同平台上的确定性，也可能让页面、RSS 与 Sitemap 随冷启动时机变化。

首次故障还暴露了 404 HTML 会继承一小时边缘缓存。Worker 已改为只缓存成功 HTML，404 与其他 HTML 错误使用 `no-store`，避免瞬时错误被边缘放大。
